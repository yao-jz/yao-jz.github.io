    git学习笔记

    ##git指令的简单介绍

    1. 创建一个git：
      在当前项目文件夹下使用git init（当前项目文件夹可以不是空的）
      在远程服务器里（储存项目）的文件夹下使用git –bare init
      –bare叫裸仓库，作为中央仓库而存在，不能成为工作的环境
      在指令的init后面可以加上这个库的名字，比如repo.git
    2. 在本地工作区（working tree）修改文件，然后使用git add <filename>或者git add *来将本地修改好的文件送入缓冲区，再使用git commit -m “message”来提交到HEAD里，最后git push origin即可
      1. 注：这里的origin是自己制定的远程仓库的名字，在push之前使用git remote add origin root@host:/path/来制定origin是哪个远程仓库位置
      2. 使用git clone root@host:/path/来克隆repo的时候，会将项目文件夹克隆到当前文件夹下

    ##git 的配置

    1. 本Mac的system gitconfig 位置：/opt/local/etc/gitconfig
    2. 在git的config里储存了系统、用户、仓库的配置信息，可以修改提交使用的姓名、邮箱、自定义的快捷键、默认的编辑器。三者config文件分别储存的位置为$(prefix)/etc/gitconfig、~/.gitconfig、当前仓库/.git/config
    3. 使用的一些指令：
      git config user.name yjz git config --global user.name yjz//修改这个用户的gitconfig git config --system user.name yjz//修改这个系统的gitconfig git config --global user.email aaa@126.com git config --global core.editor vim git config --global alias.st status//注意这里的命令不需要在前面加git，到时候直接用git st命令就可以了
    4. 注意，当前仓库config优先级>用户优先级>系统优先级

    项目的保存

    1. git可以忽略一些文件或者目录，不保存他们的更改
    2. git add的一些操作
      1. add只是将工作目录中的变化的文件送入了缓存区，告诉git想要在下一次提交时包含这个文件的更新，在使用git commit之前文件都没有被记录
      2. git add <file>/<directory>将file中或某文件夹的更改加入缓存
      3. git add -i交互式缓存
      4. 当我们修改好之后，使用git commit提交到项目历史里去
      5. 每次修改一个文件都要使用git add
      6. 可以add 路径
    3. 缓存区：当前的工作目录与项目历史之间的缓冲区。作用：可以将自己修改的内容按照逻辑划分，将相关的更改添加到缓存，一份一份提交，这样有利于追踪bug，有利于回滚项目的历史版本。（而不是将上次commit之后做的所有更改一并提交）
      1. 举个例子：你在今天对这个项目的不同模块进行了修改，但是你可以通过使用缓冲区，提交多次commit，每次commit只包含一个模块的内容，这样的话有利于以后检查bug/看历史版本的时候更加有条理性。
      2. git不会区分缓存的更改来自于新文件还是原来仓库中已有的文件
    4. git commit
      1. 提交的快照是项目的安全版本，提交到本地仓库，不会和其他git仓库有交互
      2. 已经跟踪过的文件是指使用git add的文件
      3. git commit或git commit -m "message"或git commit -a。最后一个时提交一份包含工作目录所有更改的快照，只包含跟踪过的文件的更改
      4. 在commit的message中，约定俗成的格式是：第一行用50个以内的字符总结，空一行，然后再详细阐述具体的更改

    ##仓库的状态

    1. git status
      1. 使用这个命令可以查看 哪些被缓存了，哪些没缓存，哪些没有追踪
      2. 未追踪的文件可以设置被完全忽略，只需要将路径放在一个项目根目录的.gitignore文件里，单独写成一行
        1. 示例：*.cpp~
    2. git log命令
      1. 显示已经提交的快照，作用于提交的项目历史
      2. 会显示提交唯一的标识ID（如commit c5083f3f018c1b55824de6e24908ad14a81ef5d9）
      3. 可以自定义git log的输出方式
        1. git log默认格式，超过一屏可以用空格滚动，q退出
        2. git log -n <limit>用limit限制提交的数量，只显示最近的limit个提交
        3. git log --oneline将每个提交压缩到一行
        4. git log --stat再输出哪些文件被更改了，哪些行被更改了
        5. git log -p显示每个提交全部的差异，最详细的视图
        6. git log --author="<pattern>"搜索指定作者的提交，可以使用正则表达式
        7. git log -grep="<pattern>"搜索提交信息匹配pattern的提交
        8. git log <since>..<until>现实只发生在since和until之间的提交，二者可以是提交ID（分别对应提交ID的前半段和后半段）、分支名、HEAD或是其他引用。（HEAD总是指向当前的提交）
        9. git log <file>特定文件的提交
        10. git log --graph --decorate --oneline还有一些有用的选项。--graph 标记会绘制一幅字符组成的图形，左边是提交，右边是提交信息。--decorate 标记会加上提交所在的分支名称和标签。--oneline 标记将提交信息显示在同一行，一目了然。
        11. 使用表示提交的父节点的相对引用。比如HEAD3表示当前提交的回溯三个节点的提交
        12. 以上的选项可以使用在同一个命令当中

    ##检出之前的提交

    1. git checkout
      1. 将保存的快照加载到我们的开发机器上
      2. 作用有：检出文件（查看某个特定文件的旧版本，而其他文件不变），检出提交（使工作目录与这个提交完全匹配，查看项目之前的状态，而不改变当前的状态），检出分支
      3. git checkout master回到master分支
      4. git checkout <commit> <file>查看文件之前的版本。它将工作目录中的文件变成中那个文件的拷贝，并将它加入缓存区
      5. git checkout <commit>更新工作目录中的所有文件
      6. 检出之前的提交时只读操作，这时候HEAD指向一个提交。即使编辑文件也不会影响项目的当前状态，所做的一切都不会被保存到仓库中。回到master分支之后，可以使用git revert或git reset来回滚任何更改（下一节）
      7. 但是，检出文件会影响项目当前的状态。旧版本变成了[需要提交的更改]。如果不想保留旧的版本，用git checkout HEAD <file>来返回去

    回滚错误的修改

    1. git revert
      1. 生成了撤销了commit引入的更改的新提交，然后应用到当前分支，但是之前的被撤销的提交并没有被删除。（是回滚了单独一个提交，没有移除后面的提交，然后回到项目之前的状态），但是reset则是移除了后面的提交。
      2. 没有改变项目历史
    2. git reset
      1. 危险
      2. git reset <file>从缓存区移除特定的文件，不改变工作目录，会取消这个文件的缓存，但不会覆盖更改
      3. git reset重设缓冲区，匹配最近的一次提交，不改变工作目录，可以用来重设缓存快照
      4. git reset --hard重写所有目录的更改—-清除所有未提交的更改
      5. git reset <commit>将当前分支的末端移到commit，将缓存区重设到这个提交，不改变工作目录。所有commit之后的更改会保留在工作目录中，这允许你用更干净、原子性的快照重新提交项目历史。
      6. git reset --hard <commit>缓存区和工作目录全部重设。清除了commit之后的全部提交
    3. 用一个新的提交来实现撤销
    4. git clean
      1. 将未跟踪的文件从工作目录中移除（无法撤销
      2. 注意，reset只影响被跟踪的文件，所以还需要一个命令来清理未跟踪的文件，才能恢复到之前特定提交时的状态
      3. git clean -n告诉你哪些文件将被移除
      4. git clean -f移除未被跟踪的（-f必须加），并且不会删除.gitignore里面的文件
      5. git clean -f <path>限制移除某路径下的文件
      6. git clean -df移除未跟踪的文件和目录
      7. git clean -xf移除未跟踪的和在gitignore里面的文件
